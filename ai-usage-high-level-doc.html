1 What it does

The tool is narrow on purpose: one normalized reporting surface over heterogeneous billing, quota, and token-usage providers.

💳

Balance and spend

DeepSeek, xAI, OpenRouter, Vast.ai, Exa, X API, and Nous feed the API-credit side of the report when credentials are present.

⏱️

Subscription quotas

Codex, Claude Code, and Google AI Studio render session, weekly, or per-model quota windows separately from API-credit balances.

📊

Token accounting

Providers that expose usage return cached input, uncached input, output, totals, and optional per-model rows.

🧾

Table or JSON

The same normalized data can be read as aligned terminal tables or machine-readable JSON with api and subscription branches.

🗃️

Local history

Every live fetch saves normalized snapshot rows in a local SQLite database for --history reports and provider-specific filtering.

🧯

Visible failures

Missing auth, intentional skips, timeouts, and provider-specific errors stay visible through ProviderData.meta instead of erasing the row.

2 How it works

The runtime spine is small and auditable: parse, load, fetch, normalize, persist, render.

  1. 1
    Parse operator intentargparse resolves provider selection, model detail, JSON output, history mode, and the Nous refresh helper.
  2. 2
    Load local credentialsconfig.load_credentials reads ~/.hermes/.env, fallback key files, Hermes auth state, and provider-specific OAuth files.
  3. 3
    Build provider adaptersThe registry constructs only the requested Provider subclasses, preserving the registered provider order for output.
  4. 4
    Fetch with isolationfetch_all runs provider.fetch() calls concurrently when useful, backed by a retrying urllib HTTP client and total timeout.
  5. 5
    Normalize provider dataProvider modules convert raw HTTP, CLI, or local-file responses into ProviderData, TokenData, extras, and metadata.
  6. 6
    Save snapshot rowsSnapshotDB writes normalized balance, spend, and token columns to ~/.hermes/ai-usage.db; Codex accounts get account-qualified keys.
  7. 7
    Render the reportrender_table, render_json, render_history, or render_history_json produces the operator-facing output.

3 The mental model

Think of ai-usage as a narrow reporting hub: provider-specific adapters on the right, one normalized local report on the left.

Each provider keeps its quirks inside its adapter; the CLI, renderer, and history database consume one normalized shape.

6 The local data boundary

The repo documents credential locations and normalized fields, but real credentials and raw provider payloads stay outside committed artifacts.

The dashed paths are documentation rules: credentials are read locally, normalized snapshots are stored locally, and raw provider payloads are not the committed source of truth.
